Getting Started

What Can the Companion Do for You?

Here are the kinds of situations where the Companion makes your work easier and more consistent.

📨

A user reports a problem you haven't seen before

"Grades aren't showing up for students in BIOL 1110"

  1. Open the Diagnose tab and type (or paste) the issue description
  2. If you have a Moodle URL, paste it — the Companion automatically extracts the course ID, module type, and activity
  3. The Companion searches the knowledge base, finds relevant documentation, and walks you through a structured investigation
  4. It suggests specific checks for you to perform in Moodle and questions to ask the instructor
Result: Instead of guessing, you have a systematic path to the root cause — and documentation backing up your diagnosis.
🔄

You're working a ticket you've seen variations of before

"Quiz won't let students reattempt after first try"

  1. Start a diagnosis — the Companion automatically checks past cases your team has resolved
  2. If a similar case exists, it surfaces the previous resolution steps so you don't reinvent the wheel
  3. The more cases your team saves, the smarter the Companion gets at connecting new issues to known solutions
Result: Team knowledge compounds over time. A fix discovered by one person helps everyone on the team.
✉️

You need to reply to a ticket in TeamDynamix

Crafting a clear, helpful response for an instructor or student

  1. After diagnosing the issue, click Draft Reply
  2. Choose your audience (instructor, student, or admin) and tone (professional, friendly, or brief)
  3. The Companion generates a reply you can copy directly into TDX, edit as needed, and send
Result: Professional, consistent replies in seconds — no more staring at a blank response box.
📦

An instructor says "something is wrong with my course"

Vague reports with no specific details

  1. Ask the instructor (or do it yourself) to download a .mbz backup of the course
  2. Upload it in the Diagnose tab — the Companion analyzes the course structure and runs automated health checks
  3. It detects issues like: completion tracking gaps, gradebook inconsistencies, generic activity names, misconfigured settings
Result: You find configuration problems that a user would never think to report, without needing admin access to their course.

Methodology

The Diagnostic Flow

The Companion follows a three-phase methodology. A mode badge in the interface shows you which phase is active.

Phase 1

Explore

Gather information. Suggest checks for you to perform. Ask clarifying questions.

Phase 2

Diagnose

Analyze evidence. Rank possible causes by likelihood. Cross-reference knowledge base.

Phase 3

Resolve

Provide fix steps. Draft reply for the user. Save the case for team learning.

Why this matters: Support conversations often jump straight to solutions. The Companion resists this — it investigates first, because users typically report symptoms, not root causes. A "grades aren't showing" report could be a gradebook setup issue, a completion tracking problem, a group override, or six other things.

What happens behind the scenes

  1. Knowledge base search — Your message is matched against Moodle documentation, TRU FAQs, internal guides, and past cases
  2. URL parsing — Any moodle.tru.ca URLs are automatically decoded to extract course ID, module type, and activity
  3. Context assembly — Relevant documentation, uploaded files, and past case matches are assembled into context
  4. Structured response — The AI generates a response that follows the diagnostic methodology, citing specific documentation when available
  5. Confidence signaling — The Companion tells you when it's drawing from documented knowledge vs. general Moodle expertise

The Diagnose Tab

Walking Through a Diagnosis

The Diagnose tab is where you spend most of your time. Here's what a typical session looks like.

Step 1 — Describe the Issue

Type or paste the support issue. The more context you include, the better the diagnosis.

You
An instructor reports students can't see their grades in BIOL 1110. The gradebook shows grades on her end but students see dashes. Here's the course: https://moodle.tru.ca/course/view.php?id=12345
SC
I've identified this as BIOL 1110 (course ID: 12345). This is a common gradebook visibility issue. Before diagnosing, I need to narrow down the cause. Could you check a few things?

1. In Gradebook setup, what is the aggregation method for the course total?
2. Under Course administration → Grades → User report, are there any items showing a lock icon?
3. Are the affected students in a specific group?
Tip: You can paste Moodle URLs directly into your message. The Companion automatically extracts the course ID, module type, and activity — saving you from having to describe the location manually.

Step 2 — Add Context (Optional)

The toolbar below the message box has buttons to attach additional context:

📦 .mbz Course Backup

Upload a Moodle backup file. The Companion analyzes the full course structure, activity settings, and completion configuration. Also runs automated health checks for common issues.

📸 Screenshot

Upload a screenshot of the problem. The Companion can read and analyze images — useful for error messages, unexpected layouts, or "it looks like this" reports.

📄 Saved HTML Page

Save a Moodle page with Cmd+S (Mac) or Ctrl+S (Windows), then upload the HTML file. The Companion extracts settings and configuration details from the page.

Step 3 — Follow the Investigation

Respond to the Companion's questions as you gather information. You'll see the mode badge change as the conversation progresses:

You
The aggregation is "Natural" and there are no lock icons. But I noticed the "Course total" column has a visibility icon that looks hidden. The affected students are all in Group B.
SC
That narrows it down significantly. Two likely causes:

1. Hidden course total (most likely): If the course total column visibility is set to "Hidden", students see dashes even though grades exist. The instructor can see them because the "hidden" setting doesn't apply to teachers.

2. Group override: If there's a group override on any grade items specific to Group B, it could affect visibility.

To fix: Go to Gradebook setup → Edit → Edit settings on the Course total row. Under "Grade category", check that "Hidden" is unchecked.

Step 4 — After Resolution

Once the issue is resolved, a prompt appears with your next options:

💾 Save as Case

Add a title, tags, and difficulty rating. The full conversation is saved so your team can learn from it later.

✉️ Draft Reply

Generate a reply for TeamDynamix. Choose audience (instructor, student, admin) and tone (professional, friendly, brief).

📤 Export

Download the session as Markdown for documentation or training materials.

🔗 Share

Copy a link so a colleague can view the full conversation — useful for handoffs or second opinions.

📖 Draft KB Article

Have the Companion draft a knowledge base article about the issue, ready for review and publishing.

The Search Tab

Knowledge Base

The Search tab lets you browse and search all indexed documentation using natural language — not just keywords.

How to Search

1

Type a natural question

Instead of searching for "quiz attempts", you can ask "how do I let students retake a quiz?" — the semantic search understands meaning, not just keywords.

2

Filter by source

Use the sidebar to narrow results to specific sources: Moodle Docs, TRU FAQs, internal guides, or resolved cases.

3

Jump to diagnosis

Found something relevant? Click "Diagnose this issue" on any search result to start a diagnostic conversation with that documentation pre-loaded as context.

The knowledge base grows automatically. Every time your team saves a resolved case, it gets indexed and becomes searchable. Over time, the Companion builds a library of TRU-specific solutions that no generic Moodle documentation can match.

The Cases Tab

Case Tracking & Analytics

Cases are the team's institutional memory. Every resolved issue that gets saved makes the whole team stronger.

Saving Cases

When you save a case from the Diagnose tab, you're asked for:

  • Title — A short description (e.g., "Gradebook total hidden from students in BIOL 1110")
  • Tags — Keywords for filtering later (e.g., gradebook, visibility, grades)
  • Difficulty — How hard was this to solve? (helps identify areas where the team needs training)
  • Resolution — The fix, in your own words (auto-filled from the conversation)

Browsing & Filtering Cases

1

Search by text

The search bar uses full-text search across case titles, descriptions, and resolutions.

2

Filter by tag

Click any tag pill to filter cases. Tags are shown as coloured pills on each case card and as a filter bar at the top.

3

View full details

Click a case card to see the complete conversation, resolution, and metadata.

Analytics Dashboard

Switch to the Analytics sub-tab for aggregate insights:

Weekly Trends

A 12-week bar chart showing case volume over time. Spot busy periods and seasonal patterns.

Difficulty Distribution

See how many issues are easy vs. hard. Identifies areas where the team may need additional training or documentation.

Top Modules

Which Moodle modules generate the most support issues? Quiz, Assignment, Gradebook? Data-driven prioritization.

Common Tags

A tag cloud showing the most frequent issue categories. Click any tag to jump to filtered cases.

For reporting: Use the Export CSV button to download all case data. This works in Excel and can be imported into TeamDynamix for management reporting.

Tips

Getting the Most Out of the Companion

Do Include context upfront

"Instructor in BIOL 1110 reports students can't see quiz grades. URL: moodle.tru.ca/mod/quiz/view.php?id=98765" gives much better results than "grades not showing".

Do Answer its questions

The Companion asks follow-up questions for a reason. Performing the checks it suggests and reporting back gives it the data it needs to diagnose accurately.

Do Save resolved cases

This is how the knowledge base grows. Even "simple" fixes are worth saving — they save time for the next person who encounters the same issue.

Do Use specific tags

Tags like gradebook, quiz-attempts, completion-tracking are more useful than generic tags like moodle or problem.

Avoid Treating it as a search engine

One-word queries like "gradebook" won't trigger the diagnostic methodology. Describe the actual problem or question you're investigating.

Avoid Skipping the explore phase

If the Companion asks you to check something, it's building evidence for a diagnosis. Jumping ahead to "just tell me the fix" produces less reliable results.

Remember: The Companion is a diagnostic partner, not an oracle. It works best when you bring your expertise (you know TRU, Moodle admin, and the instructor) and it brings structured investigation, documentation lookup, and pattern recognition from past cases.

Self-Hosted Setup

Installation Guide

The Companion is available as a hosted service. If you need to run your own instance, follow these steps. No prior command-line experience needed.

Before you start: You need an Anthropic API key for the AI features. Get one at console.anthropic.com (requires an account with billing). Cost is roughly $0.01–0.05 per conversation.

Mac Installation

1

Open Terminal

Press Cmd + Space, type Terminal, press Enter.

2

Install Homebrew (if needed)

Check first: brew --version. If it shows a version, skip to step 3. Otherwise:

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
3

Install Python & Node.js

Check versions: python3 --version (need 3.11+) and node --version (need 18+). Install any missing:

brew install python node
4

Download and set up the project

cd ~/Desktop
git clone https://github.com/blamb/moodle-support-companion.git
cd moodle-support-companion/backend
pip3 install -r requirements.txt
cd ../frontend
npm install
5

Set your API key and start the backend

cd ../backend
export ANTHROPIC_API_KEY="your-key-here"
python3 -m uvicorn app.main:app --port 8000 --reload

Leave this window open. You should see Uvicorn running on http://127.0.0.1:8000.

6

Start the frontend (new Terminal window)

Open a new Terminal window (Cmd + N):

cd ~/Desktop/moodle-support-companion/frontend
npm run dev

Open http://localhost:5173 in your browser.

Done! You should see the Companion with the LT&I branding and three tabs.

Windows Installation

1

Open PowerShell

Press the Windows key, type PowerShell, click Windows PowerShell.

2

Install Python

Check: python --version. If 3.11+, skip to step 3. Otherwise, download from python.org/downloads.

Critical: During installation, check "Add Python to PATH" at the bottom of the installer. Then close and reopen PowerShell.
3

Install Node.js

Check: node --version. If 18+, skip to step 4. Otherwise, download from nodejs.org (LTS version). Close and reopen PowerShell after install.

4

Download and set up the project

cd ~\Desktop
git clone https://github.com/blamb/moodle-support-companion.git
cd moodle-support-companion\backend
pip install -r requirements.txt
cd ..\frontend
npm install
If pip isn't recognized, try pip3 instead.
5

Set your API key and start the backend

cd ..\backend
$env:ANTHROPIC_API_KEY="your-key-here"
python -m uvicorn app.main:app --port 8000 --reload

Leave this window open.

6

Start the frontend (new PowerShell window)

cd ~\Desktop\moodle-support-companion\frontend
npm run dev

Open http://localhost:5173 in your browser.

Done! You should see the Companion with the LT&I branding and three tabs.

Troubleshooting

"No module named app" when starting the backend
Make sure you are in the backend folder. Run pwd (Mac) or Get-Location (Windows) to check.
API key or authentication errors
Run the export (Mac) or $env: (Windows) command again in the same terminal window where you start the backend.
"Cannot connect to the backend"
The backend server isn't running. Check the first Terminal window and restart it if needed (step 5).
Blank page after loading
Make sure you ran npm install in the frontend folder and that npm run dev is running in the second terminal window.
pip or npm command not found
Close and reopen Terminal/PowerShell after installing Python or Node.js. On Mac, try pip3. On Windows, make sure you checked "Add to PATH".

Technical Details

Architecture

For those interested in the technical underpinnings.

┌─────────────────────────────────┐
│  React + Tailwind + TypeScript  │  ← Frontend
└───────────────┬─────────────────┘
                │ SSE streaming
┌───────────────┴─────────────────┐
│          FastAPI Backend         │  ← API server
├──────────┬──────────┬───────────┤
│ ChromaDB │ SQLite   │ Claude    │
│ vectors  │ cases    │ API       │
│ (384d)   │ + FTS5   │ (Sonnet)  │
└──────────┴──────────┴───────────┘

Backend

Python / FastAPI with SSE streaming. Uvicorn ASGI server.

Vector Store

ChromaDB with sentence-transformer embeddings (384 dimensions). Heading-aware chunking.

Case Database

SQLite with FTS5 full-text search. Resolved cases auto-indexed for RAG retrieval.

AI

Anthropic Claude API with vision support. Custom diagnostic methodology prompt.

Frontend

React 19, TypeScript, Tailwind CSS v4, Vite. Custom hooks for conversation lifecycle.

Knowledge Pipeline

Ingests MediaWiki HTML, WordPress WXR, DOCX, and Moodle page exports. Deduplication by file hash.
